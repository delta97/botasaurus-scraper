A **recipe** is a small, declarative description of a browser flow. It's the
currency of the whole studio: AI runs produce recipes, the extension records
recipes, suites group recipes, schedules fire recipes, and datasets feed recipes.

## Anatomy of a recipe

```yaml
version: 1
name: example-lead-form
description: Fill out a lead form and capture the confirmation
variables:
  - { name: first_name, default: "John" }
  - { name: email, default: "john@example.com" }
botasaurus:
  headless: true
  block_images: true
  wait_for_complete_page_load: true
output_format: json
steps:
  - { type: navigate, url: "https://example.com/quote" }
  - { type: type, selector: "input[name='first_name']", value: "{{first_name}}" }
  - { type: type, selector: "input[name='email']", value: "{{email}}" }
  - { type: select_option, selector: "select[name='project']", value: "windows" }
  - { type: click, selector: "button[type='submit']" }
  - { type: assert, selector: "#thanks", message: "confirmation missing" }
  - { type: extract_markdown, into: confirmation }
```

| Field | Purpose |
| --- | --- |
| `name` / `description` | Human labels. |
| `variables` | Declared inputs with defaults; referenced as `{{name}}`. |
| `botasaurus` | Browser config applied at replay (see [Botasaurus Config](/reference/botasaurus-config)). |
| `output_format` | `json` (structured extracts) or `markdown` (whole page). |
| `steps` | The ordered actions. |

<Note>
The canonical form stored in the database is **JSON**; **YAML** is an export and
authoring convenience. They're interchangeable — YAML is a JSON superset — so the
import endpoint and CLI accept either.
</Note>

## Steps

Every step has a `type` and type-specific fields. The full catalogue is in the
[Recipe DSL reference](/reference/recipe-dsl); here are the ones you'll use most:

<CardGroup cols={2}>
  <Card title="navigate" icon="compass">
    `{ type: navigate, url }` — go to a page.
  </Card>
  <Card title="type" icon="keyboard">
    `{ type: type, selector, value }` — type into an input or textarea.
  </Card>
  <Card title="click" icon="arrow-pointer">
    `{ type: click, selector }` — click a link, button, or checkbox.
  </Card>
  <Card title="select_option" icon="list-check">
    `{ type: select_option, selector, value }` — choose a dropdown option.
  </Card>
  <Card title="extract_markdown / extract_text" icon="file-lines">
    `{ ..., into: key }` — store page content in the run result. No AI.
  </Card>
  <Card title="assert" icon="circle-check">
    Verify the page — presence, text, attributes, element count, URL.
  </Card>
</CardGroup>

Every step also supports two universal options:

<ParamField path="optional" type="boolean" default="false">
  If `true`, a failure of this step is logged and skipped instead of aborting the
  run. Useful for cookie banners that may or may not appear.
</ParamField>

<ParamField path="selector_fallbacks" type="string[]">
  Alternate selectors tried in order if the primary one fails — the first line of
  defense against site changes, before [self-healing](/guides/self-healing) ever
  involves the AI.
</ParamField>

## Variables

Any `{{name}}` placeholder in `url`, `value`, `label`, `script`, `text_contains`,
`text_equals`, or `attribute_equals` is substituted before the browser launches.

<Warning>
An unknown variable is a **hard error raised before Chrome even starts** — you'll
never burn a browser launch on a typo. Declare every variable you reference, or
pass it in at replay time.
</Warning>

```bash
# defaults come from the recipe; --var overrides them
python -m backend.runner --recipe-id 1 --var first_name=Jane --var email=jane@x.com
```

## Robust selectors

Recipes recorded by the agent or the extension don't just store one selector —
they store a **priority-ranked** primary plus fallbacks, generated by the same
engine in both Python and the extension's JavaScript (kept byte-identical by a
shared spec and golden-fixture tests). The ladder prefers stable, human-meaningful
selectors:

```text
#stable-id  →  [name=…]  →  [data-testid] / [aria-label]  →  [placeholder=…]
           →  a[href*=…]  →  nth-of-type path (flagged "fragile")
```

Auto-generated ids (hex blobs, long digit runs, React's `:r3:`) are rejected, so
your recipes don't pin themselves to selectors that change on every deploy.

## Exporting and editing

From a recipe's page you can:

- **Export** as YAML or JSON for version control or sharing.
- **Edit** the raw JSON in the browser (validated on save).
- **Replay** with variable overrides.

```bash
curl "http://localhost:8000/api/recipes/1/export?format=yaml" -o recipe.yaml
```

<Tip>
Commit exported recipes to your repo alongside your code. They're plain text,
diff cleanly, and make your automations reviewable.
</Tip>
